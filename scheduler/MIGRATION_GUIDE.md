# Migration Guide: LocalStorage to Database

This guide explains how the scheduler app has migrated from localStorage to a PostgreSQL database with Cloudflare R2 for media storage.

## What Changed?

### Before (localStorage)

- Posts were stored in the browser's localStorage
- Media files were stored as blob URLs (temporary, browser-only)
- Data was not shared across devices/browsers
- No data persistence after clearing browser data
- Limited storage capacity (~5-10MB)

### After (Database + R2)

- Posts are stored in PostgreSQL database
- Media files are uploaded to Cloudflare R2 (permanent cloud storage)
- Data is accessible from any device/browser
- Data persists and is backed up
- Virtually unlimited storage capacity

## Breaking Changes

### API Changes

#### Repository Interface

The `PostsRepository.createPost()` method now requires a `userId` parameter:

```typescript
// Before
await repository.createPost({
  message: "Hello world",
  accountIds: ["account1"],
  media: [],
  scheduledFor: new Date(),
  status: "scheduled",
});

// After
await repository.createPost(
  {
    message: "Hello world",
    accountIds: ["account1"],
    media: [],
    scheduledFor: new Date(),
    status: "scheduled",
  },
  userId,
);
```

### Component Changes

#### SchedulePostForm

- Now uploads files to API endpoint (`POST /api/posts`)
- Files are sent as `FormData` with multipart upload
- Media URLs are generated by R2 (not blob URLs)

#### PostsList

- Now fetches posts from API endpoint (`GET /api/posts`)
- No longer uses client-side repository

### Database Schema

New tables have been added:

```sql
-- Posts table
CREATE TABLE post (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  message TEXT NOT NULL,
  scheduled_for TIMESTAMP NOT NULL,
  status TEXT DEFAULT 'scheduled',
  published_at TIMESTAMP,
  account_options JSONB,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Media files table
CREATE TABLE media_file (
  id TEXT PRIMARY KEY,
  post_id TEXT NOT NULL,
  url TEXT NOT NULL,
  type TEXT NOT NULL,
  filename TEXT NOT NULL,
  size INTEGER NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  FOREIGN KEY (post_id) REFERENCES post(id) ON DELETE CASCADE
);

-- Many-to-many relationship between posts and accounts
-- (handled by Prisma's implicit join table)
```

## Migration Steps

### For Development

1. **Update environment variables** (see `ENVIRONMENT.md`):

   ```env
   R2_ENDPOINT="https://..."
   R2_ACCESS_KEY_ID="..."
   R2_SECRET_ACCESS_KEY="..."
   R2_BUCKET_NAME="..."
   R2_PUBLIC_URL="..."
   ```

2. **Run database migrations**:

   ```bash
   yarn db:push
   ```

3. **Restart the development server**:
   ```bash
   yarn dev
   ```

### For Production

1. **Set up Cloudflare R2**:
   - Create a new R2 bucket
   - Generate API credentials
   - Configure custom domain for public access

2. **Update production environment variables**

3. **Deploy database migrations**:

   ```bash
   yarn db:push  # or db:migrate for production
   ```

4. **Deploy application**

### Data Migration

**Note**: There is no automatic migration from localStorage to database. Existing posts in localStorage will remain in the browser but won't be synced to the database.

If you need to migrate existing localStorage data:

1. **Export localStorage data**:

   ```javascript
   // Run in browser console
   const posts = JSON.parse(localStorage.getItem("social-scheduler-posts") || "[]");
   console.log(JSON.stringify(posts, null, 2));
   // Copy the output
   ```

2. **Manually recreate posts** through the UI or API

## Rollback (if needed)

If you need to temporarily rollback to localStorage:

1. Update `lib/config.ts`:

   ```typescript
   const REPOSITORY_TYPE: RepositoryType = "localStorage";
   ```

2. Restart the application

**Warning**: This will not access posts stored in the database. Only use for testing/debugging.

## Troubleshooting

### "Unauthorized" errors

- Ensure you're logged in
- Check that Better Auth is properly configured

### Media upload fails

- Verify R2 credentials in environment variables
- Check R2 bucket permissions
- Ensure R2_PUBLIC_URL is set correctly

### Database connection errors

- Verify DATABASE_URL is correct
- Ensure database is running and accessible
- Check network/firewall settings

### Posts not appearing

- Clear browser cache
- Check browser console for API errors
- Verify posts exist in database using Prisma Studio: `yarn db:studio`
